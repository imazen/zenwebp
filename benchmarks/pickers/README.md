# zenwebp picker code-heuristics

Interpretable, zero-load CART pickers for the zenwebp encoder config space — lossy (both metrics) and lossless — generated by the 2026-06-28 dual-model fan-out. The accurate MLP peers are documented here but mostly not committed (see "Why no MLP .bin").

## What's here

- `zenwebp_lossy_cart_zensim_2026-06-28.rs` — lossy picker fit against zensim quality.
- `zenwebp_lossy_cart_ssim2_2026-06-28.rs` — lossy picker fit against SSIMULACRA2.
- `zenwebp_lossless_cart_ssim2_2026-06-28.rs` — lossless picker (pure size optimization).

Each file is a standalone, auto-generated decision tree — `pick_zenwebp_lossy_heuristic(feats, zq)` or `pick_zenwebp_lossless_heuristic(feats, zq)` — that returns a sweep-cell id (an index into the dump's `cell_labels`) from the 469 zenanalyze image features and a target quality `zq` in 0..100. No model file, no runtime, no dependencies — just nested f64 comparisons that match sklearn exactly. Each file also carries a `main()` that byte-exact-verifies the tree against the `*_cases.bin` fixture on R2. These are reference artifacts; they are not wired into the crate build.

The lossless picker is metric-agnostic: lossless configs produce identical pixels, so any quality metric is degenerate and the oracle is pure minimum-bytes. The zensim and ssim2 lossless CARTs from the fan-out are byte-identical; only one is committed here.

## Two pickers, two roles

CART is the interpretable no-load code-heuristic: a hard decision tree compiled to branches, with nothing to ship but source. The MLP is the accurate ZNPR peer: a small (~205 KB f16) net that picks more sharply but needs the model file plus the zenpredict runtime. For lossless, that ordering inverts — see below.

## Measured overhead

Val mean RD overhead vs the per-row reachable oracle (imazen-26 corpus, origin even/odd split, held-out validation; lower is better):

| picker | metric | val mean overhead | notes |
|---|---|---|---|
| CART (committed)     | lossy zensim    | 4.7%  | depth-6, 64 leaves |
| CART (committed)     | lossy ssim2     | 6.1%  | depth-6, 63 leaves |
| CART (committed)     | lossless        | 1.0%  | depth-6, 49 leaves |
| MLP (not committed)  | lossy zensim    | 3.37% | bake refused, forced --allow-unsafe |
| MLP (not committed)  | lossy ssim2     | 2.65% | bake refused, forced --allow-unsafe |
| MLP (none exists)    | lossless        | 2.65% | R2 file is a duplicate of lossy ssim2 |

For reference, the per-cell GBDT teacher in each CART's own dump scores 6.0% (lossy zensim) / 7.3% (lossy ssim2) / 2.0% (lossless). On lossy, CART and MLP overheads come from different fan-out stages (cart_analysis's depth-6 tree over the 30-cell space vs the train_hybrid byte-regression student), so read the gap as indicative, not a strict head-to-head.

On lossless the picture is simpler and better: the committed depth-6 CART (1.0%) beats both its GBDT teacher (2.0%) and the supposed MLP (2.65%), so it is the best available lossless picker — no model file needed.

## Why no MLP .bin

The lossy MLP ZNPR bakes on R2 are not committed: both were forced with `--allow-unsafe` past a `BAKE REFUSED` (lossy zensim a WORST_ROW at 328% RD overhead; lossy ssim2 a PER_ZQ_TAIL p99 > 80% at zq 82 plus a WORST_ROW at 412%). Shipping an `--allow-unsafe` bake is what the picker program forbids; their fate is pending a deploy-mode (top-K-verify) gate decision in the main session.

There is no genuine lossless MLP. The `zenwebp_lossless__ssim2/train_hybrid/` artifact on R2 is byte-identical (sha256 and all sidecars) to the lossy ssim2 bake — a fan-out wiring bug where the lossless train_hybrid stage consumed the lossy dataset. The lossless picker_tree_ab dump (which fed the CART) is correct and distinct; only the train_hybrid bake was mis-wired. Either way the committed lossless CART already beats it.

## Provenance

- Source: `s3://zentrain/dualmodel-2026-06-28/zenwebp_lossy/picker_tree_ab/` (lossy zensim), `.../zenwebp_lossy__ssim2/...` (lossy ssim2), `.../zenwebp_lossless__ssim2/...` (lossless). Copied verbatim, renamed.
- Generator: `zenmetrics/scripts/train/cart_analysis.py`, depth swept 3..16, depth-6 codegen'd.
- Split: imazen-26 origin even/odd (`zenmetrics/scripts/picker/origin_split.py`) — train even origins, val {1,3,5}, test {7,9}; no rendition leakage.
- Cells: 30 lossy configs, 9 lossless configs. The id->config table is the dump `cell_labels` in `zenwebp_*_meta.json` (inside the `dump_*.tar` on R2).
- Dominant split features (GBDT permutation importance): pixel count, padded-pixel size buckets, quant survival (lossy) / min and max dimension and patch fraction (lossless).

## Regenerating

```
# fetch and extract a dump tar from R2, then:
python3 zenmetrics/scripts/train/cart_analysis.py \
  --dump-dir <extracted>/val --codec-tag zenwebp_lossy --eval-split val \
  --codegen-depth 6 --codegen-out zenwebp_lossy_cart.rs
# (use --codec-tag zenwebp_lossless for the lossless picker)
```
