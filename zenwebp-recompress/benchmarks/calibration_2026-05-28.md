# Calibration sweep — per-content-class reencode tables (2026-05-28)

Replaces the preliminary 3-photo fit in `src/calibration/data.rs` with a
disciplined, per-content-class calibration. Tables live in
`src/calibration/calib_tables.rs` (AUTO-GENERATED by
`zwr-calibrate/fit_calibration.py`).

## What the tables predict

For a lossy-VP8 source of effective quality `eff_q`, content class `class`,
re-encoded at libwebp quality `q`:

- `{CLASS}_CUM[anchor][q]` — cumulative zensim-A vs the *original* (the quality
  the recompressed output lands at).
- `{CLASS}_RATIO[anchor][q]` — `output_len / source_len`.
- `{CLASS}_SOURCE_CUM = (a, b)` — the source's *own* cumulative `a·q + b` (the
  ceiling no recompression can beat; the NoOp check).

The router (`best_reencode`) interpolates over `eff_q` and scans the q columns
for the cheapest q whose cumulative meets the target and that actually shrinks.

## Sweep design (source-informing discipline)

- **Content classes (4):** photo (CID22-512), screen (imazen web
  screenshots), line-art (zensim text-ui), mixed (aic3 originals). Per-class
  tables — screen/line-art recompress nothing like photo.
- **Size variants:** 64 / 256 / 1024 max-dim + native (capped 2048),
  Mitchell-Netravali resample, **downscale-only** (no synthetic upscales).
  Built by `zwr-calibrate/build_calib_corpus.sh`.
- **Synthetic sources:** each ref encoded at q20–100 step 2 (the input range
  the tool must handle).
- **Raw reencode-q grid:** 20–95 step 5, swept via `run_reencode_at_q`
  (non-circular — NOT the calibration-mediated path). `--reencode-qs` mode.
- **Held-out validation:** 20% of refs per class (`val/` split), never seen by
  the fit.

## Keying decision

The router keys the calibration on the **decode-based estimate**
(`estimate.rs`), which is biased **~−5** vs true q (MAE ~6.5; 73.5% within ±10,
measured on the pilot). The tables are nonetheless fit on the **TRUE** encode
quality, because:

1. Keying on the estimate starves the top anchor (q90) — the estimator rarely
   reaches 90, leaving that row empty.
2. Est-vs-true keying was measured **equivalent** (<0.5 cumulative MAE on the
   pilot): interpolation over the 30..90 anchors absorbs the −5 offset.

So: fit on true q (all anchors populated), index at runtime with the estimate
(bias absorbed). `plan()` uses the rougher header estimate (documented).

## Size dependence

Measured per the discipline (ratio = f(size)): at mid-q (source≈70,
reencode≈70) the size_ratio spread across tiny/small/medium/large buckets is
modest (~3% for photo). The per-class tables are size-averaged over the web
range; the runtime **ground-truth size guard** corrects any residual
per-image error (never ships a non-shrinking output).

## Results

Full run: 50 refs/class → ~371 size-variant refs, q20–100 step 2 sources,
reencode-q 20–95 step 5. **248,501 ok train cells, 0 errors**; 63,879 val.
(`git=e9de934`, host lilith, 2026-05-29.)

### Held-out validation (cumulative-zensim MAE, full run)

| class    | MAE  | p50  | p90   | n      |
|----------|------|------|-------|--------|
| photo    | 7.89 | 6.42 | 16.98 | 19,680 |
| screen   | 3.56 | 2.64 |  7.66 | 11,808 |
| line-art | 8.49 | 6.69 | 18.77 |  5,904 |
| mixed    | 3.98 | 3.20 |  8.80 |  3,936 |

screen/mixed predict tightly (MAE ~3.5–4). photo/line-art MAE ~8 with a heavy
p90 tail: that's **within-class content variance** — a smooth photo and a
detailed one recompress very differently at the same q, which a per-class
*mean* table can't capture. Mitigations at runtime: the ground-truth size
guard (never ships a non-shrinking output) and the `MaxIterations` budget
(measures the actual output instead of trusting the model). This is still a
large improvement over the prior photo-only 3-ref fit, which had **no**
screen/line-art table at all (it would mis-predict those classes badly).

### Size-dependence (ratio at source≈70, reencode≈70, by bucket)

| class    | tiny  | small | medium | large |
|----------|-------|-------|--------|-------|
| photo    | 0.963 | 0.953 | 0.945  | —     |
| screen   | 0.965 | 0.962 | 0.964  | 0.962 |
| line-art | —     | 0.972 | 0.976  | 0.983 |
| mixed    | 0.960 | 0.947 | 0.914  | 0.908 |

Spread across size buckets at mid-q is modest (~2% photo/screen/line-art, ~5%
mixed). The per-class tables are size-averaged over the web range; the size
guard corrects residual per-image error. (photo has no `large` bucket — CID22
sources are 512px; line-art has no `tiny` — text-ui images are already small.)

## Caveats

- **Mixed is thin** (only ~10 clean source refs available locally) — its table
  is the weakest. The size guard keeps it correct regardless; a larger mixed
  corpus is the obvious follow-up.
- Line-art refs (text-ui) are small, so they contribute few large-size
  variants; the line-art table's large-size cells lean on fewer samples.
